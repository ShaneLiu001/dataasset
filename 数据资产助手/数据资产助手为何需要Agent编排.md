# 数据资产助手为何需要 Agent 编排

## 1. 核心结论

大模型能力变强以后，数据资产助手更需要 Agent 编排，而不是更不需要。

原因在于，大模型主要解决的是“理解和生成”问题，而企业数据资产平台真正要解决的是“可靠执行、权限控制、状态跟踪、证据可查、结果可审计”的问题。

可以把两者关系理解为：

```text
大模型：负责理解用户意图、生成候选方案、解释复杂结果
Agent 编排：负责拆解任务、调度工具、控制流程、记录状态、处理异常
MCP 工具网关：负责连接元数据、血缘、稽核、数据源、审批等现有系统
业务状态中心：负责长流程跟踪、任务恢复、事件回调和审计追踪
观测评测平台：负责记录 Trace、评估效果、优化 Prompt 和工具调用
```

所以，数据资产助手不是简单问答机器人，而是一个“AI 驱动的数据资产操作入口”。大模型是大脑，Agent 编排是让大脑安全、稳定、可追踪地调用企业系统的操作系统。

## 2. 大模型擅长什么

大模型非常适合承担以下工作：

1. 理解自然语言诉求。
2. 识别用户意图和业务概念。
3. 抽取表名、字段名、指标、主题域、时间范围等槽位。
4. 根据上下文生成 SQL 草稿、稽核规则草案、元数据说明。
5. 总结血缘链路、治理建议和异常原因。
6. 把复杂平台结果转成用户能理解的自然语言解释。

例如用户说：

```text
帮我找客户收入相关的权威表，看看血缘是否可靠，然后配置每日金额非空稽核。
```

大模型可以很好地理解：

1. 用户要找“客户收入”相关数据资产。
2. 需要判断资产是否权威。
3. 需要查看血缘可靠性。
4. 最后要配置一个每日稽核规则。

这些属于理解、推理和生成能力，是大模型的强项。

## 3. 大模型不能单独兜住什么

但在企业数据资产平台里，只理解用户意图远远不够。真正落地时还必须回答以下问题：

1. 用户有没有权限查看这张表？
2. 字段是否涉及敏感信息，能不能展示样例值？
3. 查询出来的资产是否来自权威元数据系统？
4. 血缘结果来自哪个版本，是否完整可信？
5. 稽核规则创建前，是否需要用户确认？
6. 数据源登记流程当前走到哪一步？
7. 审批流、工单、采集任务的回调丢失后如何补偿？
8. 工具调用失败后是否重试，重试几次？
9. 用户关闭页面后，下次进入如何恢复任务？
10. AI 给出的建议依据是什么，后续如何审计和追责？

这些问题不是模型参数规模变大就能自然解决的，它们属于工程治理和企业级系统集成问题。

因此，不能让大模型直接自由调用业务系统，而必须通过 Agent 编排和工具网关建立可控边界。

## 4. 为什么数据资产场景尤其需要 Agent 编排

数据资产助手面对的不是单一问答场景，而是跨平台、跨流程、跨权限的复杂业务场景。

涉及系统包括：

1. 元数据管理平台。
2. 数据地图。
3. 数据血缘平台。
4. 统一数据源平台。
5. 数据稽核平台。
6. 数据安全平台。
7. 审批流和工单系统。
8. 调度任务平台。
9. 指标平台。
10. 本体语义服务。

这些系统各自有权限、参数、状态、错误码和流程规则。大模型可以决定“应该做什么”，但真正执行时必须由确定性的 Agent 编排和工具系统来完成。

推荐链路如下：

```text
用户自然语言
  -> 大模型识别意图和槽位
  -> 主 Agent 生成执行计划
  -> 子 Agent 执行专业任务
  -> MCP 工具网关校验权限、参数和幂等
  -> 调用元数据、血缘、稽核、数据源等业务系统
  -> 状态中心记录任务、事件、结果和审计
  -> 大模型汇总证据并解释给用户
```

这种模式既发挥了大模型的理解能力，也保证了企业系统执行的安全性和稳定性。

## 5. 典型例子

用户输入：

```text
帮我找客户收入相关表，看看血缘是否可靠，然后配置每日金额非空稽核。
```

如果只靠大模型直接回答，最多能给出一个建议：

```text
建议查找客户收入宽表，并为收入金额字段配置非空校验。
```

但生产级数据资产助手必须完成完整闭环：

1. 识别“客户收入”对应的业务概念、本体实体和指标口径。
2. 搜索候选表、字段和指标。
3. 判断哪张表是权威资产。
4. 查询表级和字段级血缘。
5. 判断上游来源是否可靠、下游影响范围是什么。
6. 校验当前用户是否有权限查看该资产。
7. 找到金额字段和分区字段。
8. 读取历史数据质量分布和规则模板。
9. 生成稽核规则草案。
10. 展示规则内容、调度周期、阈值和告警级别。
11. 等待用户确认。
12. 调用稽核平台创建规则。
13. 记录规则 ID、操作者、时间、依据、traceId。
14. 后续支持查询规则运行结果。

这里面，大模型主要负责第 1、3、9 和结果解释；其余步骤都需要 Agent 编排、工具调用、权限控制、状态管理和审计机制。

## 6. Agent 编排解决的关键问题

### 6.1 任务拆解

用户的一句话往往包含多个动作，例如查表、查血缘、配置稽核、提交治理。主 Agent 需要把复杂诉求拆成明确步骤，再调度不同子 Agent。

```text
查表问数 Agent
  -> 血缘分析 Agent
  -> 数据稽核 Agent
  -> 权限与风险 Agent
```

### 6.2 工具调用

数据资产助手必须调用真实平台能力，而不是只生成文本。

例如：

1. 调用元数据搜索接口。
2. 调用血缘查询接口。
3. 调用数据源登记接口。
4. 调用稽核规则创建接口。
5. 调用元数据审核发布接口。

这些接口应该统一封装为 MCP 工具，由工具网关负责鉴权、参数校验、幂等和错误码转换。

### 6.3 状态管理

很多数据资产任务不是一次请求就能完成。

例如数据源登记涉及：

```text
登记表单提交
  -> 直属领导审批
  -> 登记人复核
  -> 连通性检测
  -> 防火墙开通
  -> 元数据采集
  -> 数据地图更新
  -> 空间项目绑定
```

这类长流程必须进入业务状态中心，不能只保存在大模型上下文里。

### 6.4 人工确认

数据资产平台里有很多高风险操作：

1. 发布元数据。
2. 下线资产。
3. 批量修改字段说明。
4. 创建或发布稽核规则。
5. 调整安全等级。
6. 发起数据源登记和扫描任务。

Agent 编排需要区分只读查询、可恢复写操作和高风险写操作。只读查询可以自动执行，高风险操作必须展示影响范围并等待用户确认。

### 6.5 权限和安全

数据资产涉及敏感字段、权限角色、脱敏策略和审计要求。

Agent 编排必须确保：

1. 用户身份透传到工具网关。
2. 工具调用前做权限校验。
3. 敏感样例值不直接进入大模型上下文。
4. 大模型输出不得泄露无权限资产。
5. 所有写操作可追溯到具体用户。

### 6.6 可观测和可评测

企业级 AI 应用不能只看“回答像不像”，还要评估：

1. 意图识别是否准确。
2. 查表推荐是否命中权威资产。
3. 血缘解释是否基于真实证据。
4. 工具调用是否成功。
5. 稽核规则生成是否可执行。
6. 用户是否采纳 AI 建议。

因此需要 Langfuse / OpenTelemetry 记录每次会话、Prompt、模型输入输出、工具调用、检索证据、成本和延迟。

## 7. 裸用大模型与 Agent 编排的差异

| 对比项 | 裸用大模型 | Agent 编排模式 |
| --- | --- | --- |
| 定位 | 问答和生成 | 可执行的数据资产助手 |
| 数据来源 | 依赖上下文和提示词 | 调用权威业务系统 |
| 任务范围 | 单轮或短对话 | 支持长流程和多步骤任务 |
| 工具调用 | 容易失控或难审计 | 统一经过 MCP 工具网关 |
| 权限控制 | 难以保证 | 每次调用前校验权限 |
| 状态跟踪 | 依赖对话上下文 | 进入业务状态中心 |
| 失败处理 | 模型重新回答 | 重试、补偿、回调、告警 |
| 人工确认 | 需要人工约定 | 流程内置确认节点 |
| 审计追踪 | 不完整 | traceId、事件、日志全链路 |
| 适用场景 | 咨询、草稿、解释 | 查询、治理、登记、稽核、发布 |

## 8. 建设价值

建设 Agent 编排不是为了重复造大模型能力，而是为了把大模型真正接入企业数据资产平台。

它带来的价值包括：

1. 将自然语言转化为可执行的平台动作。
2. 打通元数据、血缘、稽核、数据源、安全、审批等分散能力。
3. 让复杂流程可跟踪、可恢复、可审计。
4. 让 AI 建议有证据、有置信度、有来源。
5. 降低用户使用数据资产平台的门槛。
6. 为后续主动治理、智能稽核、自动文档、智能问数打基础。

最终目标不是做一个“会聊天”的助手，而是建设一个“能安全办事”的数据资产智能操作入口。

## 9. 一句话总结

大模型解决“听懂人话”和“生成建议”，Agent 编排解决“把建议变成企业系统里安全、可靠、可追踪的动作”。

因此，数据资产助手必须做 Agent 编排。
