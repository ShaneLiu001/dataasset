# 本体平台切入建设方案

## 一、结论先行

在已经具备元数据管理、统一数据源、数据质量稽核、数据血缘、数据权限、安全扫描、统一指标平台、数据建模平台和数据标准平台的前提下，本体平台建设仍然有意义，但切入优先级需要调整：

> 一期先做元数据中文名自动补全和元数据备注自动生成，二期再基于补全后的语义元数据支撑报表平台 AI 问数。

由于组织架构原因，当前不适合建设独立标签平台；同时，现有统一指标平台沉淀的内容更多偏指标元数据，并不一定已经形成可直接支撑 AI 问数的完整业务语义层。因此，本体平台的首要价值应先落在“把技术元数据补成业务可理解的语义元数据”上：

- 表中文名、字段中文名自动补全。
- 表备注、字段备注自动生成。
- 字段业务含义、字段角色、枚举说明、使用限制自动建议。
- 补全结果给出置信度、依据和人工确认入口。
- 人工确认结果沉淀为后续本体、指标语义和 AI 问数的训练语料。

在此基础上，再补齐报表问数所需的语义上下文：

- 用户问题中的业务对象是什么。
- 问题对应哪些报表、指标、维度和筛选条件。
- 指标口径是否清楚，是否存在同名异义。
- 报表字段、指标字段、模型字段和底层物理字段如何关联。
- 时间、区域、组织、客户、产品等常见条件应如何解释。
- 当前用户能不能看、应该看汇总值还是明细值。
- 生成答案时应引用哪个权威口径和哪条数据链路。

一句话定位：

> 本体平台的一期抓手是元数据语义补全，最终目标是成为报表平台 AI 问数的语义底座；它不是标签平台，也不是替代现有数据治理平台的新门户。

## 二、需求背景

当前提出本体平台建设，核心原因不是“平台能力清单缺一项”，而是报表平台问数质量差。但从落地顺序看，如果底层表、字段、报表和指标的中文名、备注、业务含义本身不完整，直接做 AI 问数会很容易继续出现找错表、找错字段、解释不清的问题。

因此，建议把一期目标前置为：

> 先补全元数据中文名和备注，形成可被人审、可被检索、可被 AI 消费的语义元数据底座。

报表平台 AI 问数质量差，通常不是单纯大模型能力不足，而是缺少结构化语义支撑。典型问题包括：

| 问题 | 表现 | 本质原因 |
|---|---|---|
| 找不到正确报表 | 用户问“本月收入情况”，系统不知道该用哪个报表 | 报表没有绑定业务主题、指标、维度和适用场景 |
| 找错指标 | 把合同金额、收入金额、回款金额混用 | 指标口径、业务定义、适用报表没有语义约束 |
| 维度解释错误 | “区域”到底是客户区域、机构区域还是销售区域 | 缺少业务对象和维度归属 |
| 时间口径错误 | “本月”按创建时间、交易时间还是确认时间统计不清楚 | 缺少指标时间口径和报表默认时间字段 |
| 同名字段歧义 | 多张表都有 `amount`、`status`、`org_id` | 缺少字段到业务属性的映射 |
| 权限不清 | 问数结果可能越权或被过度拦截 | 权限、安全等级和业务对象未联动 |
| 答案不可解释 | 用户不知道答案来自哪个报表、哪个指标、哪个口径 | 缺少可追溯语义链路 |

因此，建设重点不是先做通用本体建模工具，而是先建立“元数据语义补全能力”，再围绕报表问数逐步沉淀“报表语义本体”。

## 三、现有能力与缺口判断

当前平台能力已经比较全，但这些能力更多是分散存在。

| 已有能力 | 当前价值 | 对 AI 问数的不足 |
|---|---|---|
| 元数据管理 | 有表、字段、任务、报表等技术元数据 | 技术元数据不能直接表达业务问题含义 |
| 统一数据源 | 有统一数据接入和连接管理 | 解决数据在哪里，不解决该用什么数据 |
| 数据质量稽核 | 有质量规则和稽核结果 | 质量结果没有转化为问数可信度提示 |
| 数据血缘 | 有表、字段、任务、指标链路 | 血缘偏技术链路，缺少业务语义解释 |
| 数据权限 | 有授权和审批能力 | 权限没有和问数意图、报表场景充分联动 |
| 安全扫描 | 能识别敏感字段和风险资产 | 安全结果缺少业务对象和组合风险解释 |
| 统一指标平台 | 沉淀指标名称、字段、来源、口径等元数据 | 仍需补业务对象、适用场景、问法、维度约束 |
| 数据建模平台 | 有逻辑模型、物理模型、主题域和分层 | 模型未必能被 AI 直接理解和选择 |
| 数据标准平台 | 有术语、字段标准、编码和值域 | 标准与报表、指标、用户问法之间缺少映射 |

真正缺口是：

1. **元数据语义缺口**

   很多表、字段只有英文名、缩写名、技术注释或历史备注，缺少稳定的中文名、业务含义、使用说明和限制条件。AI 问数前需要先让底层资产具备可理解的语义描述。

2. **报表语义缺口**

   报表缺少业务主题、适用问题、核心指标、默认维度、默认时间口径、适用人群、不可回答范围等语义描述。

3. **指标语义缺口**

   指标平台已有指标元数据，但 AI 问数还需要知道指标对应的业务对象、业务口径、可分析维度、可用报表、同义问法和冲突口径。

4. **业务对象缺口**

   系统缺少客户、产品、合同、交易、机构、账户等统一业务对象，导致 AI 难以判断字段和指标属于哪个业务语境。

5. **问法映射缺口**

   用户自然语言中的“收入”“规模”“客户数”“本月”“华东”“重点客户”等，没有稳定映射到报表、指标、维度、筛选条件。

6. **答案解释缺口**

   AI 即使给出答案，也缺少“来自哪个报表、采用哪个指标、按什么口径、有什么权限和质量风险”的解释链路。

## 四、目标定位

建议把本体平台定位为“元数据语义补全 + 报表问数语义支撑平台”。一期先服务元数据管理和数据资产盘点，二期服务报表平台 AI 问数，再逐步扩展到指标、权限、质量和数据开发助手。

分三层建设：

| 层次 | 建设内容 | 目标 |
|---|---|---|
| 元数据语义层 | 表中文名、字段中文名、表备注、字段备注、字段业务含义 | 先让技术元数据变成业务可理解的语义资产 |
| 报表语义层 | 报表主题、报表指标、维度、筛选项、适用问法、默认口径 | 让 AI 知道用户问题该找哪张报表 |
| 业务对象层 | 客户、产品、合同、交易、机构等对象及属性关系 | 让 AI 理解问题中的业务实体和条件 |
| 问数服务层 | 问题解析、报表匹配、指标匹配、维度映射、口径解释、权限校验 | 直接提升 AI 问数准确率和可解释性 |

第一阶段不建议追求大而全的企业级本体，而是围绕高价值表、字段、报表字段和指标字段，先把中文名和备注补全做准。

## 五、总体架构建议

建议采用“元数据补全先行 + 报表问数牵引 + 现有平台供数 + 本体平台供语义”的架构。

```text
现有平台
  元数据 / 报表 / 指标元数据 / 建模 / 标准 / 血缘 / 质量 / 权限 / 安全扫描
        |
        v
语义汇聚层
  汇聚报表字段、指标元数据、模型字段、标准术语、血缘、质量、权限和安全结果
        |
        v
元数据语义补全层
  表中文名 / 字段中文名 / 表备注 / 字段备注 / 业务含义 / 置信度 / 生成依据
        |
        v
报表语义本体层
  报表主题 / 业务对象 / 指标语义 / 维度语义 / 时间口径 / 问法样例 / 约束规则
        |
        v
AI 问数语义服务层
  意图识别 / 报表匹配 / 指标映射 / 维度筛选 / 权限校验 / 口径解释 / 结果溯源
        |
        v
报表平台 AI 问数
  问题理解 / 推荐报表 / 生成查询 / 返回答案 / 展示依据 / 反馈纠错
```

关键原则：

- 报表平台仍负责问数入口、交互体验和结果展示。
- 元数据管理平台仍负责元数据维护入口、审核流程和正式发布。
- 指标平台继续作为指标元数据的重要来源，但不假设它已经完整解决 AI 问数语义。
- 元数据、标准、模型、血缘、质量、权限、安全平台继续作为各自领域权威源。
- 本体平台负责把这些分散信息组织成 AI 可消费的语义上下文。

## 六、核心建设内容

### 1. 元数据中文名和备注自动补全

一期优先建设元数据语义补全能力，先解决“表字段看不懂、备注不完整、业务含义不一致”的问题。

补全对象建议包括：

| 对象 | 自动补全内容 | 示例 |
|---|---|---|
| 数据表 | 表中文名、表备注、业务主题、数据粒度 | 客户基础信息表；存储客户主数据基础档案 |
| 字段 | 字段中文名、字段备注、字段业务含义、字段角色 | 客户编号；客户在主数据系统中的唯一标识 |
| 报表字段 | 展示名称、业务解释、对应指标或维度 | 收入金额；按财务确认口径统计的收入 |
| 指标字段 | 指标中文名、口径摘要、适用维度 | 活跃客户数；统计周期内发生有效交易的客户数 |
| 枚举字段 | 枚举含义、值域说明、默认解释 | 客户状态：1 有效，0 无效 |

自动补全需要综合以下上下文：

- 表名、字段名、字段类型、长度、是否为空。
- 原有表注释、字段注释、历史备注。
- 所属系统、库、Schema、数仓分层、主题域。
- 数据标准、术语、编码和值域。
- 数据建模平台中的逻辑模型和物理模型。
- 指标平台中的指标元数据。
- 报表字段、报表展示名、报表筛选项。
- SQL 血缘、Join 条件、上下游任务。
- 安全扫描结果、质量规则、权限范围。
- 同名字段、相似字段、历史人工确认结果。

生成结果必须包含：

```text
MetadataCompletionResult
  object_type
  object_id
  generated_cn_name
  generated_remark
  business_meaning
  confidence
  evidence
  risk_warning
  review_status
```

关键原则：

1. 不覆盖人工已确认内容，只对空值、低质量备注或待补全项生成候选。
2. 所有候选结果必须进入人工确认或抽样审核流程。
3. 对缩写严重、上下文不足、证据冲突的字段，不强行生成确定性描述。
4. 备注不能只重复中文名，必须说明业务含义、取值、用途或限制。
5. 生成结果要保留证据，便于后续沉淀为本体属性、指标语义和问数规则。

### 2. 报表语义管理

报表是当前 AI 问数的最直接抓手。建议优先对高频报表做语义增强。

每张重点报表建议补齐：

| 语义项 | 示例 | 作用 |
|---|---|---|
| 报表业务主题 | 经营收入分析、客户增长分析、风险监控 | 帮助 AI 判断问题所属主题 |
| 适用问题 | 查收入、查客户数、查交易趋势 | 帮助 AI 匹配自然语言问题 |
| 不适用问题 | 不用于财务确认收入、不用于监管口径 | 避免误用报表 |
| 核心指标 | 营业收入、合同金额、回款金额、活跃客户数 | 建立报表和指标关系 |
| 可分析维度 | 时间、区域、机构、产品、客户类型 | 限定可筛选、可分组范围 |
| 默认时间口径 | 交易日期、确认日期、入账日期 | 避免时间字段误选 |
| 默认统计粒度 | 日、月、季度、年 | 避免聚合粒度错误 |
| 权限范围 | 部门级、机构级、个人级、集团级 | 支撑问数权限判断 |
| 质量提示 | 数据延迟、缺失风险、口径调整 | 支撑答案可信度提示 |

### 3. 指标语义增强

现有统一指标平台沉淀的指标更多偏元数据。本体平台不重复建设指标平台，而是在其基础上补充面向问数的语义。

建议为高频指标补齐：

- 指标业务定义。
- 同义词和常见问法。
- 反义或易混指标。
- 统计口径和过滤条件。
- 默认时间字段。
- 可分析维度。
- 适用报表。
- 不适用场景。
- 指标负责人和口径版本。
- 底层字段、模型、血缘和质量状态。

示例：

```text
指标：收入金额
常见问法：收入、营收、营业收入、本月收入、收入趋势
易混指标：合同金额、回款金额、开票金额
默认时间口径：财务确认日期
适用报表：经营收入分析报表、收入趋势报表
不适用场景：不用于实时交易流水统计
```

### 4. 业务对象管理

业务对象是连接报表、指标、字段和用户问题的骨架。

优先建设对象：

| 优先级 | 业务对象 | 与问数的关系 |
|---|---|---|
| P0 | 客户、产品、机构、交易、合同 | 高频筛选和分组条件 |
| P1 | 账户、订单、渠道、员工、项目 | 支撑更多专题报表 |
| P2 | 风险事件、营销活动、供应商、报表 | 支撑专项分析和治理扩展 |

业务对象需要维护：

- 对象名称、别名、业务定义。
- 对象属性，例如客户编码、客户名称、客户等级、所属区域。
- 对象关系，例如客户发生交易、客户签订合同、产品产生收入。
- 对象属性与报表字段、模型字段、标准字段的映射。
- 对象属性与指标维度、筛选项的映射。

### 5. 问法样例与语义映射

AI 问数要提升，必须沉淀真实问法。

建议从报表平台历史问数日志、用户搜索词、报表访问记录、运营常见问题中抽取问法样例。

问法样例结构：

```text
QuestionExample
  question_text
  intent_type
  matched_report
  matched_metric
  matched_dimensions
  matched_filters
  time_range
  expected_answer_type
  review_status
```

示例：

```text
问题：今年华东区域收入趋势怎么样？
意图：趋势分析
报表：经营收入分析报表
指标：收入金额
维度：月份、区域
筛选：区域 = 华东
时间：今年
答案类型：趋势图 + 摘要说明
```

### 6. 约束规则与口径解释

本体平台需要记录 AI 问数时必须遵守的规则。

常见规则：

- “收入”默认使用财务确认收入，不使用合同签约金额。
- “本月”默认按自然月，除非报表另有定义。
- “客户区域”不能和“机构区域”混用。
- “集团口径”必须使用集团级汇总报表。
- 查询敏感明细时需要权限，无法直接返回个人敏感信息。
- 数据质量异常时，答案必须提示风险。

这些规则不只是文档说明，而要能被问数服务调用。

## 七、AI 问数服务流程

建议本体平台为报表平台提供一个问数语义解析链路。

```text
用户问题
  -> 识别业务意图
  -> 识别业务对象、指标、维度、时间、筛选条件
  -> 匹配候选报表
  -> 匹配候选指标和字段
  -> 校验权限、安全、质量状态
  -> 生成语义解释和查询约束
  -> 返回报表平台执行查询或推荐报表
  -> 展示答案、口径、来源和风险提示
  -> 用户反馈进入语义优化闭环
```

输出给报表平台的结果建议包括：

```text
SemanticParseResult
  intent
  business_objects
  metrics
  dimensions
  filters
  time_range
  matched_reports
  confidence
  permission_check
  quality_warning
  explanation
  fallback_question
```

当置信度不足时，不应强行回答，而应反问或推荐候选报表。例如：

```text
你说的“收入”可能指财务确认收入、合同金额或回款金额，请选择口径。
```

## 八、分阶段落地路线

### 第一阶段：元数据中文名和备注自动补全

周期建议：1 到 2 个月。

目标：

- 先补齐高价值表、字段、报表字段和指标字段的中文名与备注。
- 降低元数据人工维护成本。
- 为后续报表问数建立可被 AI 消费的底层语义资产。

重点工作：

1. 选择优先补全范围：核心指标来源表、高频报表依赖表、核心模型表、数据标准覆盖表、高价值资产。
2. 建立中文名和备注生成模板，包括表中文名、表备注、字段中文名、字段备注、业务含义、使用限制。
3. 接入元数据、数据标准、建模、指标、报表、血缘、质量、安全扫描等上下文。
4. 生成候选中文名和备注，并输出置信度、生成依据和风险提示。
5. 建立人工确认流程，确认结果回写元数据管理平台。
6. 沉淀已确认结果，形成后续业务对象、指标语义、问法映射的基础语料。

交付物：

- 元数据补全范围清单。
- 中文名和备注生成规范。
- 元数据补全服务。
- 补全结果确认页面或确认流程。
- AI 候选值、人工最终值、置信度和依据记录。
- 补全采纳率和准确率评估报告。

### 第二阶段：报表问数问题诊断与高频报表语义补齐

周期建议：2 到 3 个月。

目标：

- 找出报表平台问数质量差的主要原因。
- 选择高频、高价值、问题集中的报表做试点。
- 在已补全元数据基础上建立报表语义模板。

重点工作：

1. 收集历史问数失败案例。
2. 按失败原因分类：找错报表、找错指标、时间口径错误、维度歧义、权限问题、质量问题。
3. 选取 10 到 30 张高频报表作为试点。
4. 补齐报表主题、适用问题、核心指标、维度、时间口径、权限范围和质量提示。
5. 建立报表问法样例库。

交付物：

- 问数失败原因分析报告。
- 高频报表语义清单。
- 报表语义模板。
- 问法样例库。
- 试点报表问数准确率基线。

### 第三阶段：指标语义和业务对象映射

周期建议：2 到 4 个月。

目标：

- 将现有指标元数据增强为可问数的指标语义。
- 建立核心业务对象及其与报表、指标、字段的映射。

重点工作：

1. 梳理高频指标和易混指标。
2. 补齐指标常见问法、同义词、易混口径、默认时间字段、可用维度。
3. 建立客户、产品、机构、交易、合同等核心业务对象。
4. 建立对象属性到报表字段、模型字段、标准字段的映射。
5. 建立指标到报表、指标到业务对象、指标到维度的映射。

交付物：

- 高频指标语义清单。
- 易混指标口径对照表。
- 核心业务对象模型。
- 对象属性映射表。
- 指标语义映射表。

### 第四阶段：问数语义服务接入报表平台

周期建议：2 到 4 个月。

目标：

- 将本体语义能力服务化，接入报表平台 AI 问数链路。
- 让问数结果可解释、可追溯、可纠错。

重点功能：

1. 问题解析服务。
2. 报表匹配服务。
3. 指标和维度映射服务。
4. 权限、安全、质量校验服务。
5. 口径解释和来源追溯服务。
6. 用户反馈纠错服务。

交付物：

- AI 问数语义 API。
- 报表平台接入规范。
- 语义解析结果页或解释组件。
- 问数反馈闭环。
- 试点效果评估报告。

### 第五阶段：从报表语义扩展为企业级本体

触发条件：

- 元数据补全结果已稳定沉淀为语义资产。
- 报表问数准确率显著提升。
- 多个业务域和多类报表已经沉淀稳定语义。
- 数据开发助手、资产搜索、权限申请、质量治理也开始消费语义服务。

此时再扩展：

- 图谱化本体建模。
- 多业务域本体管理。
- 版本管理和变更审批。
- 语义冲突检测。
- 复杂关系推理。
- 面向数据资产平台的对象级资产地图。

## 九、与现有平台的集成关系

### 1. 与报表平台

报表平台是本体建设的二期重点消费方。一期通过元数据补全先改善底层语义质量，二期再把已确认的语义元数据接入报表问数链路。

本体平台提供：

- 报表语义描述。
- 问法匹配。
- 指标、维度、筛选条件解析。
- 口径解释。
- 权限和质量提示。
- 结果来源追溯。

### 2. 与统一指标平台

指标平台提供指标元数据，包括指标名称、编码、来源、口径描述、维度、负责人等。

本体平台补充：

- 常见问法。
- 同义词和易混指标。
- 指标适用报表。
- 指标默认时间口径。
- 指标与业务对象、字段、模型的语义映射。

### 3. 与元数据管理平台

元数据管理平台是本体建设的一期消费方，提供表、字段、任务、报表等技术元数据，并承接补全结果的确认和正式回写。

本体平台补充：

- 表中文名、字段中文名、表备注、字段备注候选。
- 字段业务属性。
- 报表字段含义。
- 字段与指标、维度、筛选项的关系。
- AI 问数使用依据。

### 4. 与数据标准平台

数据标准平台提供术语、字段标准、编码和值域。

本体平台补充：

- 标准术语与用户问法的映射。
- 标准字段与报表字段、模型字段的映射。
- 枚举值和值域在问数中的解释。

### 5. 与数据建模平台

数据建模平台提供逻辑模型、物理模型和主题域。

本体平台补充：

- 模型实体到业务对象的映射。
- 模型字段到对象属性的映射。
- 模型与报表、指标的关系。

### 6. 与质量、权限、安全平台

这些平台继续作为权威能力来源。

本体平台补充：

- 问数时的数据质量提示。
- 权限不足时的解释和可选替代报表。
- 敏感指标和敏感明细的问数限制。
- 安全扫描结果与业务对象、报表字段的关联。

## 十一、试点场景建议

### 场景一：核心表字段中文名和备注补全

试点对象：

- 核心指标来源表。
- 高频报表依赖表。
- 核心模型表。
- 数据标准覆盖较多的表。
- 业务 Owner 明确、便于审核的高价值资产。

重点解决：

- 英文字段名、缩写字段名难以理解。
- 表备注和字段备注缺失或质量差。
- 同类字段中文名不一致。
- AI 问数缺少底层字段语义。
- 人工盘点元数据成本高。

输出结果：

- 表中文名。
- 表备注。
- 字段中文名。
- 字段备注。
- 字段业务含义。
- 置信度和生成依据。
- 人工确认结果。

### 场景二：收入类报表问数

典型问题：

- 本月收入是多少？
- 今年各区域收入趋势怎么样？
- 华东区域收入同比增长多少？

重点解决：

- 收入、合同金额、回款金额、开票金额的口径区分。
- 时间口径选择。
- 区域维度归属。
- 报表匹配和指标匹配。

### 场景三：客户类报表问数

典型问题：

- 本月新增客户数是多少？
- 重点客户有哪些？
- 各行业客户数量分布如何？

重点解决：

- 客户对象定义。
- 新增客户、活跃客户、重点客户口径。
- 客户等级、行业、区域等维度映射。

### 场景四：交易类报表问数

典型问题：

- 今日交易金额是多少？
- 各产品交易笔数排名如何？
- 异常交易趋势怎么样？

重点解决：

- 交易时间和入账时间区分。
- 交易金额、交易笔数、成功率指标口径。
- 产品、渠道、机构维度映射。

## 十三、效果评估

一期建议先用元数据补全效果衡量价值，二期再用报表问数效果衡量本体平台对 AI 问数的支撑价值。

一期元数据补全指标：

| 指标 | 说明 |
|---|---|
| 表中文名补全率 | 试点范围内表中文名自动生成覆盖比例 |
| 字段中文名补全率 | 试点范围内字段中文名自动生成覆盖比例 |
| 备注补全率 | 表备注和字段备注自动生成覆盖比例 |
| 人工采纳率 | AI 候选结果被人工直接采纳或轻微修改后采纳的比例 |
| 高置信度准确率 | 高置信度候选经人工确认后正确的比例 |
| 人工节省时长 | 相比纯手工盘点节省的录入和解释时间 |
| 依据可解释率 | 候选结果能展示有效来源证据的比例 |
| 回写成功率 | 已确认结果成功回写正式元数据的比例 |

二期报表问数指标：

| 指标 | 说明 |
|---|---|
| 报表匹配准确率 | 用户问题是否匹配到正确报表 |
| 指标匹配准确率 | 是否选对指标和口径 |
| 维度筛选准确率 | 是否正确识别时间、区域、组织、客户、产品等条件 |
| 有效回答率 | 能直接回答或给出正确追问的比例 |
| 口径解释覆盖率 | 答案是否能展示指标、报表、时间口径和来源 |
| 用户采纳率 | 用户是否接受答案或继续使用推荐报表 |
| 反馈修正率 | 用户纠错后是否能沉淀为语义规则 |

## 十四、风险与规避

| 风险 | 表现 | 规避建议 |
|---|---|---|
| 仍按标签平台思路推进 | 组织上无法承接，建设目标偏离问数 | 明确不建独立标签平台，标签只作为字段属性或安全属性存在 |
| 元数据补全生成幻觉 | 中文名或备注看似合理但无证据 | 必须展示依据和置信度，低置信度进入人工补充 |
| AI 覆盖人工内容 | 已确认元数据被自动覆盖 | 默认只补空值和待补全项，人工确认内容优先 |
| 备注质量低 | 只重复字段中文名，没有业务含义 | 制定备注生成规范，要求包含含义、用途、取值或限制 |
| 指标平台被误认为已解决语义 | AI 仍然找错指标、误解口径 | 把指标平台视为指标元数据来源，本体补问法、场景和约束 |
| 本体模型过大 | 建模周期长，短期无改善 | 先围绕元数据补全和高频报表建最小语义集 |
| 只做文档不做服务 | 元数据平台或报表平台无法调用语义能力 | 一期建设补全服务，二期建设问数语义 API |
| AI 强答 | 置信度不足仍输出错误答案 | 低置信度时反问、推荐候选报表或要求选择口径 |
| 缺少反馈闭环 | 同类错误反复出现 | 把用户纠错沉淀为问法样例、指标别名和约束规则 |

## 十五、最终建议

当前本体平台建设建议按以下路径推进：

```text
第 1 步：先做元数据中文名和备注自动补全
  对核心表、字段、报表字段、指标字段生成中文名、备注、业务含义和依据

第 2 步：建立补全确认和回写闭环
  由人工确认 AI 候选结果，正式回写元数据管理平台，并沉淀确认样本

第 3 步：再分析报表平台问数失败案例
  搞清楚问数差是报表匹配、指标口径、维度歧义、权限还是质量问题

第 4 步：补齐高频报表语义
  为重点报表维护主题、适用问题、核心指标、维度、时间口径和限制范围

第 5 步：增强指标语义
  在现有指标元数据基础上补充问法、别名、易混指标、适用报表和约束规则

第 6 步：建设核心业务对象
  用客户、产品、机构、交易、合同等对象串起报表、指标、维度和字段

第 7 步：接入报表平台 AI 问数链路
  通过语义 API 返回问题解析、报表匹配、指标映射、权限质量提示和口径解释

第 8 步：用反馈持续优化本体
  把用户纠错、人工确认、问数失败案例沉淀为问法样例和语义规则

第 9 步：再扩展企业级本体能力
  当报表问数跑通后，再支撑数据开发助手、资产搜索、影响分析和治理自动化
```

一句话总结：

> 先用本体和 AI 做元数据中文名、备注、业务含义补全，把技术元数据变成语义元数据；再用这些语义资产支撑报表平台 AI 问数。
