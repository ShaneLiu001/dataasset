# 数据开发助手如何基于本体论快速查找元数据

## 一、目标

数据开发助手的核心能力，不只是帮用户按字段名搜索表，而是能理解用户的业务意图，并快速定位到正确的数据资产。

例如用户问：

```text
我要开发一个客户收入分析任务，应该用哪些表？
```

传统元数据搜索只能匹配“客户”“收入”等关键词，结果可能返回大量无关表。引入本体论后，助手可以理解：

- “客户”是业务实体 Customer
- “收入”可能对应指标 contract_revenue 或 finance_revenue
- “客户收入分析”需要客户、合同、财务收入等实体关系
- 需要优先返回权威来源表、主数据表、事实表和已认证指标

最终目标是让数据开发人员更快完成：

- 找表
- 找字段
- 找指标
- 找口径
- 找关联关系
- 找血缘
- 找样例 SQL
- 判断数据是否可用、可信、可访问

## 二、总体思路

可以把本体论作为数据开发助手的“语义导航层”。

整体链路如下：

```text
用户自然语言问题
  -> 意图识别
  -> 业务概念抽取
  -> 本体概念映射
  -> 关系扩展
  -> 元数据召回
  -> 权威性排序
  -> 口径与血缘解释
  -> 返回表、字段、指标、关联方式和样例 SQL
```

本体论的作用，是把用户的自然语言映射到企业统一语义模型，再由语义模型定位到底层元数据。

## 三、元数据查找的核心对象

数据开发助手需要查找的对象不应只包括表和字段，建议至少覆盖以下几类：

| 类型 | 示例 | 作用 |
|---|---|---|
| 业务实体 | 客户、合同、项目、产品、组织 | 理解业务对象 |
| 业务属性 | 客户名称、客户等级、所属区域 | 映射字段 |
| 指标 | 合同收入、回款金额、客户数 | 统一口径 |
| 数据表 | crm_customer、contract、finance_revenue | 定位物理数据 |
| 字段 | cust_id、contract_id、confirmed_amount | 开发取数 |
| 关系 | 客户签订合同、合同产生收入 | 生成 Join 路径 |
| 血缘 | 指标来源、报表来源、加工链路 | 判断可信度 |
| 规则 | 有效客户、重点客户、今年 | 解释过滤条件 |
| 权限 | 敏感级别、可访问角色 | 判断是否可用 |
| 样例 | 典型 SQL、典型问法 | 辅助开发 |

## 四、基于本体论的快速查找机制

### 1. 建立业务概念索引

先把本体中的实体、属性、指标、规则建立成统一概念索引。

示例：

```json
{
  "concept_id": "Customer",
  "label": "客户",
  "aliases": ["客户信息", "客户主数据", "客户档案", "企业客户"],
  "type": "entity",
  "description": "与公司发生业务关系的组织或个人",
  "mapped_tables": ["crm_customer", "mdm_customer"],
  "mapped_fields": ["crm_customer.cust_id", "crm_customer.cust_name"]
}
```

这样用户输入“客户档案”“客户主数据”“企业客户”时，都能映射到 Customer。

### 2. 建立字段语义映射

字段不能只靠字段名匹配，要建立“字段 -> 业务属性”的映射。

示例：

```text
crm_customer.cust_id -> Customer.customer_id -> 客户编码
crm_customer.cust_name -> Customer.customer_name -> 客户名称
finance_revenue.confirmed_revenue_amount -> Metric.contract_revenue -> 合同收入
```

这样用户问“客户名称字段在哪张表”，助手可以直接返回权威字段，而不是靠 `name`、`cust` 这类模糊匹配。

### 3. 建立关系图谱

把实体之间的关系、表之间的 Join 关系、指标来源关系放到图结构中。

示例：

```text
Customer --signs--> Contract --generates--> FinanceRevenue
Customer.customer_id = Contract.customer_id
Contract.contract_id = FinanceRevenue.contract_id
```

当用户问“客户收入”时，助手可以从 Customer 和 Revenue 两个概念出发，自动找到中间关系 Contract，形成可执行的 Join 路径。

### 4. 建立指标口径索引

指标是数据开发中最容易出错的对象。要把指标定义、公式、时间口径、维度和来源字段统一索引。

示例：

```json
{
  "metric": "contract_revenue",
  "label": "合同收入",
  "formula": "sum(finance_revenue.confirmed_revenue_amount)",
  "time_field": "finance_revenue.confirmed_date",
  "dimensions": ["客户", "区域", "行业", "项目", "产品"],
  "source_table": "finance_revenue",
  "certified": true
}
```

当用户问“今年收入”时，助手应优先使用已认证指标和指标定义中的时间字段。

### 5. 建立向量索引和关键词索引

建议采用混合检索：

- 关键词检索：适合表名、字段名、编码、系统名等精确查找
- 向量检索：适合自然语言、别名、描述、口径说明等语义查找
- 图检索：适合实体关系、Join 路径、血缘追溯

推荐检索策略：

```text
第一步：关键词召回表、字段、指标
第二步：向量召回业务概念、说明文档、口径规则
第三步：图谱扩展相关实体、关系和血缘
第四步：按权威性、相关性、可访问性排序
```

## 五、查询处理流程

### 场景 1：找表

用户问题：

```text
客户主数据在哪张表？
```

处理流程：

1. 抽取概念：客户主数据
2. 映射本体：Customer
3. 查找映射表：crm_customer、mdm_customer
4. 判断权威来源：优先返回 mdm_customer
5. 返回字段、主键、更新频率、负责人和使用建议

返回示例：

```text
建议优先使用 mdm_customer。

原因：
- 该表是客户主数据的权威来源
- 主键为 customer_id
- 包含客户名称、行业、区域、客户等级等核心属性
- 下游 CRM 和收入分析任务均引用该客户编码
```

### 场景 2：找字段

用户问题：

```text
客户等级字段在哪里？
```

处理流程：

1. 抽取属性：客户等级
2. 映射本体：Customer.customer_level
3. 查找来源字段：crm_customer.customer_level
4. 返回字段解释、枚举值和可用表

返回示例：

```text
客户等级对应字段为 crm_customer.customer_level。

字段说明：
- 中文名：客户等级
- 枚举值：A、B、C、D
- 所属实体：客户
- 可用于重点客户识别规则
```

### 场景 3：找指标

用户问题：

```text
合同收入怎么算？
```

处理流程：

1. 抽取指标：合同收入
2. 映射本体：Metric.contract_revenue
3. 查找公式：sum(finance_revenue.confirmed_revenue_amount)
4. 查找时间口径：finance_revenue.confirmed_date
5. 查找维度：客户、区域、项目、产品、组织
6. 返回口径、来源字段和注意事项

### 场景 4：找 Join 关系

用户问题：

```text
客户表怎么关联收入表？
```

处理流程：

1. 抽取实体：客户、收入
2. 映射本体：Customer、FinanceRevenue
3. 图谱查找路径：Customer -> Contract -> FinanceRevenue
4. 返回关联字段和样例 SQL

返回示例：

```sql
SELECT
  c.customer_id,
  c.customer_name,
  SUM(r.confirmed_revenue_amount) AS contract_revenue
FROM mdm_customer c
JOIN contract ct
  ON c.customer_id = ct.customer_id
JOIN finance_revenue r
  ON ct.contract_id = r.contract_id
GROUP BY
  c.customer_id,
  c.customer_name;
```

### 场景 5：找血缘

用户问题：

```text
这个收入指标来自哪些表？
```

处理流程：

1. 抽取指标：收入指标
2. 映射到具体指标：contract_revenue
3. 查找 lineage
4. 返回来源字段、加工任务、下游报表

## 六、排序规则

元数据召回后，需要做排序。否则助手会返回一堆结果，用户仍然不知道该用哪个。

建议排序权重：

| 排序因素 | 说明 |
|---|---|
| 语义相关度 | 与用户问题的匹配程度 |
| 权威性 | 是否为主数据、认证指标、官方口径 |
| 使用热度 | 被任务、报表、用户引用的频率 |
| 数据新鲜度 | 最近更新时间、同步频率 |
| 数据质量 | 完整性、准确性、异常率 |
| 权限可用性 | 当前用户是否有权限访问 |
| 血缘可信度 | 是否来源清晰、加工链路完整 |
| 命名匹配度 | 表名、字段名、别名是否匹配 |

推荐公式：

```text
score =
  语义相关度 * 0.30
  + 权威性 * 0.25
  + 使用热度 * 0.15
  + 数据质量 * 0.10
  + 新鲜度 * 0.10
  + 权限可用性 * 0.10
```

## 七、助手返回结果格式

为了让数据开发人员能直接使用，返回结果建议固定为以下结构：

```text
1. 推荐使用的数据资产
2. 推荐原因
3. 核心字段
4. 关联关系
5. 指标口径或业务规则
6. 数据血缘
7. 权限与风险提示
8. 样例 SQL
9. 相关文档或负责人
```

示例：

```text
推荐使用 finance_revenue 表计算合同收入。

推荐原因：
- 是合同收入指标的认证来源表
- 金额字段为 confirmed_revenue_amount
- 时间口径为 confirmed_date
- 可通过 contract_id 关联合同表，再通过 customer_id 关联客户表

注意：
- 不建议直接使用 contract.signed_amount 作为收入
- signed_amount 是签约金额，不等同于财务确认收入
```

## 八、系统架构建议

建议数据开发助手采用以下架构：

```text
前端对话入口
  -> 意图识别服务
  -> 本体语义解析服务
  -> 混合检索服务
      -> 关键词索引
      -> 向量索引
      -> 图数据库
      -> 元数据库
  -> 排序与权限过滤
  -> LLM 生成解释和 SQL
  -> 审计与反馈
```

各组件职责：

- 本体语义解析服务：把自然语言映射到实体、属性、指标、规则
- 关键词索引：处理表名、字段名、系统名、编码精确搜索
- 向量索引：处理业务描述、别名、口径说明语义搜索
- 图数据库：处理实体关系、Join 路径、血缘追溯
- 元数据库：存储表、字段、指标、权限、质量、热度等结构化元数据
- LLM：负责问题理解、结果解释、SQL 生成和多轮追问

## 九、落地优先级

建议分三期建设。

### 第一期：能找准

目标是让助手能准确找到表、字段、指标。

建设内容：

- 业务术语表
- 核心实体模型
- 字段语义映射
- 指标口径库
- 关键词 + 向量混合检索

### 第二期：能解释

目标是让助手能说明为什么推荐这些数据。

建设内容：

- 指标口径解释
- 表字段说明
- 权威来源标识
- 数据质量标签
- 权限标签
- 负责人和文档链接

### 第三期：能开发

目标是让助手能辅助生成开发资产。

建设内容：

- Join 路径推荐
- 样例 SQL 生成
- 数据血缘追溯
- 任务模板推荐
- 数据校验规则生成
- 口径冲突提醒

## 十、关键结论

用本体论做元数据快速查找，核心不是把所有文档丢给大模型，而是先建立一层稳定的业务语义模型。

本体论负责回答：

- 用户说的业务词是什么意思
- 这个业务词对应哪些表和字段
- 指标应该按什么口径计算
- 表和表之间应该怎么关联
- 哪些数据是权威来源
- 哪些数据当前用户可以访问

数据开发助手负责把这些能力包装成自然语言交互，让开发人员少翻文档、少问人、少试错，更快找到可信、可用、可解释的数据资产。
